# 数据收集说明（实验报告可直接引用）

> 本文档与仓库内 `data/raw/ai_knowledge.json` 一致，用于支撑「数据收集」章节的**数据来源、方法、步骤、质量评估与统计展示**。  
> 说明：本课程项目采用**小规模领域知识图谱**，数据为**基于公开权威/可查资料的人工抽取与结构化**，重点在于验证「收集—表示—存储—应用—评测」全流程；**未使用**某一整套现成的公开知识图谱数据集（如 CN-DBpedia 全量）作为直接输入，也**未**编写大规模网络爬虫自动抓取网页。

---

## 一、实验目的（可与课程模板一致）

本次数据收集旨在为构建「中国先进人工智能技术」领域知识图谱获取**高质量、可解释、可复现**的结构化知识，为后续知识表示、图谱存储（Neo4j）、问答与评测奠定基础。具体包括：

1. 明确数据来源类型：以**公开政策文本、行业公开信息、百科类公开条目、学术/技术常识**为主，覆盖结构化（表格字段）、半结构化（条目+属性）与非结构化（段落摘要→属性）三类信息的**人工抽取结果**。
2. 采用**可追溯的收集与标注流程**：每条实体与关系均对应可查的公开信息来源类型（见下文「来源分类」），保证可复核。
3. 将数据整理为统一的 JSON Schema，并通过导入脚本写入 Neo4j，满足后续实验对**格式一致性与可查询性**的要求。

## 三、数据收集方法（来源 + 工具）

### 3.1 数据来源（如何写进报告——不是「随便写」）

本图谱数据属于**领域自建知识库**，其知识内容对齐以下**公开可查**的信息类型（建议在报告中写成「来源类别」，并各举 1～2 个代表）：

| 来源类型 | 含义 | 在本项目中的体现（举例） |
|----------|------|---------------------------|
| **政策与规划文本（结构化/半结构化）** | 政府公开发布的战略文件 | 如《新一代人工智能发展规划》作为政策类实体，及其与「人工智能」的推动关系 |
| **企业与产品公开信息（半结构化）** | 企业官网、新闻稿、产品发布页等公开材料 | 如「文心一言」「通义千问」「昇腾AI」等与机构、技术分支的关系 |
| **技术常识与公开综述（非结构化→结构化）** | 教材、百科、综述性条目，经人工摘要为「描述」字段 | 如「Transformer」「大语言模型」等定义性描述 |
| **公开技术史事实（可核对）** | 代表性模型、论文、产品的时间节点 | 实体中 `year` 字段用于支撑时间维检索与答辩讲解 |

**重要表述建议（答辩/报告可用）：**

> 本项目数据**并非**从某一公开知识图谱数据集整库下载，也**未**采用全自动爬虫抓取海量网页；而是采用**「公开来源 + 人工抽取 + 统一 Schema」**的半自动化数据工程路径：先确定领域与实体类型体系，再依据公开可查资料填写实体属性与关系，形成**可解释、可扩展**的小规模 gold 数据，用于课程实验全流程验证。

### 3.2 收集工具

| 工具 | 用途 |
|------|------|
| 浏览器 | 检索政策原文、企业公开介绍、百科条目等 |
| 文本编辑器 / VS Code / PyCharm | 编辑与校验 `ai_knowledge.json` |
| Python + `scripts/01_import_data.py` | 将 JSON 批量导入 Neo4j |
| Neo4j Browser | 可视化抽查节点与关系是否导入正确 |

**未使用工具（如实写即可）：** 大规模分布式爬虫框架（Scrapy 等）、第三方付费封闭数据库。

## 四、数据收集步骤与质量评估

### 4.1 不同来源数据的处理方式与整合方式

| 数据形态 | 处理方式 | 如何整合进统一图谱 |
|----------|----------|---------------------|
| **政策/新闻等段落文本（非结构化）** | 人工阅读 → 抽取「政策名称—推动—技术领域」等三元组 → 写入 `relations` | 与 `entities` 中已有实体对齐；无则新增实体节点 |
| **百科/教材条目（半结构化）** | 提炼「名称、类型、定义、时间、地域」→ 映射为实体字段 `name/type/description/year/country` | 所有实体统一为 `Entity` 节点（Neo4j 中标签），属性一致 |
| **产品—机构—技术关系（半结构化）** | 依据公开「研发/发布」信息，标注 `研发、实现、属于` 等关系类型 | 关系统一为 `RELATION` 边，`type` + `description` 双字段说明语义 |
| **质量控制** | 双人交叉核对或自查清单（见 4.2） | 通过 JSON Schema 约束字段；导入后在 Neo4j 中抽样查询验证 |

**整合逻辑（一句话写进报告）：**

> 各类来源经人工抽取后，统一落地为 **「实体表 + 关系表」** 两层结构（JSON），再通过同一导入脚本写入 Neo4j，实现多来源知识在**同一本体模式**下的融合。

### 4.2 准确性、完整性、一致性评估（有标准、可写进报告）

| 维度 | 评估方法 | 本项目结论（可按实际修改） |
|------|----------|------------------------------|
| **准确性** | 对关键实体（政策名、知名产品、机构名）与公开网页/原文标题核对；关系类型是否符合常识 | 采用公开可查的常识与公开材料表述；**年份类字段**为教学用近似值，若课程要求极高精度，可注明「以进一步人工校对为准」 |
| **完整性** | 是否覆盖「技术—模型—产品—机构—应用—政策—基础设施」链条 | 当前为**小规模演示集**，覆盖主要链条但**不声称**覆盖全行业全量知识 |
| **一致性** | 实体 `name` 是否在 `relations` 的 `source/target` 中完全同名引用；类型枚举是否统一 | JSON 内实体名称与关系端点**一一对应**；`type` 使用统一英文枚举（Technology/Organization 等），便于程序处理 |

### 4.3 数据数量与结构展示（与仓库一致）

**统计（以 `data/raw/ai_knowledge.json` 为准）：**

| 指标 | 数值 |
|------|------|
| 实体数量 | **35** |
| 关系数量 | **43** |
| 实体类型数 | **10** 类 |

**实体类型分布（便于报告画图或做表）：**

| 类型（英文枚举） | 中文含义（报告可写） |
|------------------|----------------------|
| Technology | 技术方向/方法 |
| Algorithm | 算法/架构 |
| Model | 模型 |
| Product | 产品/系统 |
| Organization | 机构/企业 |
| Person | 人物 |
| Application | 应用场景 |
| Hardware | 硬件/芯片平台 |
| Policy | 政策/规划 |
| Infrastructure | 基础设施 |

**关系类型（部分示例，报告中可写「关系类型体系」）：**

属于、基于、应用、研发、提出、支撑、依赖、扩展、实现、改进、推动、提供、研究 等。

**存储结构（Neo4j）：**

- 节点标签：`Entity`
- 节点属性：`id, name, type, description, year, country, reference`（`reference` 为参考文献/来源说明，支持报告追溯）
- 关系类型：`RELATION`
- 关系属性：`type`（语义关系，如「属于」）、`description`（自然语言说明）、`reference`（该关系依据的公开资料说明）

**参考文献字段维护：** 实体与关系的 `reference` 可由 `scripts/03_enrich_references.py` 批量生成初稿后，在 `data/raw/ai_knowledge.json` 中人工校对增删；再执行 `01_import_data.py` 写入 Neo4j。
